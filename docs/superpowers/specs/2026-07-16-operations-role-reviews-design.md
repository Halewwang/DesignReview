# 运营角色与补充评价设计

## 目标

在现有设计师、管理员角色之外增加“运营”角色。运营可以查看设计师创建的全部审核任务，并针对具体提交轮次追加补充评价；运营评价不改变任务状态、AI 评分或最终结论。

同时为生产环境配置管理员专用访问口令 `emke666`，保持管理员权限与设计师、运营权限隔离。

## 非目标

- 不恢复已取消的人工复审状态流，不重新启用 `operation_review` 或 `director_review` 任务状态。
- 运营不能执行通过、退回、归档、删除、撤回、重新提交或 AI 重试。
- 运营不能创建审核任务、修改任务名称、选择 Frame、上传图片或修改系统设置。
- 运营评价不替代 AI 审核，也不参与分数计算。

## 角色与登录

客户端和服务端会话统一支持三种可登录角色：

- 设计师：使用现有 `REVIEW_ACCESS_CODE`。
- 运营：与设计师共享 `REVIEW_ACCESS_CODE`。
- 管理员：使用独立的 `REVIEW_ADMIN_ACCESS_CODE`，生产值为 `emke666`。

用户选择角色并登录后，服务端签发包含固定角色的会话。后续授权只读取服务端会话角色，不接受客户端通过请求头临时提升权限。历史客户端会话标准化逻辑接受“运营”，现有设计师和管理员会话继续有效。

## 权限矩阵

| 能力 | 设计师 | 运营 | 管理员 |
| --- | --- | --- | --- |
| 查看自己的任务 | 是 | 是 | 是 |
| 查看全部设计师任务 | 否 | 是 | 是 |
| 创建和维护自己的任务 | 是 | 否 | 是 |
| 追加运营评价 | 否 | 是 | 否 |
| 查看运营评价 | 是 | 是 | 是 |
| 管理员通过归档 | 否 | 否 | 是 |
| 修改 AI/VIS 设置 | 否 | 否 | 是 |
| 删除任意任务 | 否 | 否 | 是 |

管理员保留现有的服务端超级权限语义，但运营评价接口会显式要求“运营”角色，避免管理员操作被记录成运营评价。

## 数据模型

复用现有 `OperationReview`：

```ts
type OperationReview = {
  id: string;
  taskId: string;
  submissionRound: number;
  reviewerName: string;
  comment: string;
  focus: string;
  createdAt: string;
};
```

每次提交创建一条新记录：

- `submissionRound` 取任务当前提交轮次。
- `reviewerName` 取服务端会话姓名。
- `comment` 去除首尾空白后必须非空。
- `focus` 为可选关注点，空值保存为空字符串。
- 记录只追加，不更新或覆盖已有评价。

现有 Postgres 采用 JSONB 文档存储，`operationReviews` 已包含在数据库结构中，因此不需要数据库迁移；旧数据会继续由标准化逻辑读取。

## API 设计

恢复以下接口：

```http
POST /api/reviews/:id/operation-review
Authorization: Bearer <session-token>
Content-Type: application/json

{
  "comment": "补充评价正文",
  "focus": "可选关注点"
}
```

处理规则：

1. 只允许服务端会话角色为“运营”的用户调用。
2. 任务不存在时返回 404。
3. `comment` 为空时返回 400，错误提示为“请输入运营补充评价”。
4. 成功后追加 `OperationReview` 和审计日志，任务状态、分数、更新时间和最终结论保持不变。
5. 返回新建的评价记录，状态码为 201。
6. 设计师或管理员调用时返回 403。

现有任务详情接口继续返回按任务筛选的 `operationReviews`，并为旧记录补齐缺失的 `submissionRound`。

## 前端设计

### 登录页

- 角色下拉增加“运营”。
- 运营使用普通“访问口令”文案，并沿用设计师默认访问口令。
- 管理员继续显示独立的“管理员访问口令”提示。
- 角色切换时清除错误；从管理员切换到设计师或运营时恢复普通访问口令，从普通角色切换到管理员时清空口令。

### 工作台

- 服务端任务列表只向设计师返回其本人创建的任务，向运营和管理员返回全部任务。
- 设计师直接请求他人任务详情时返回 403；运营和管理员可以打开任意任务详情。
- 运营默认展示全部任务，`onlyMine` 为 `false`。
- 隐藏“新建审核任务”入口。
- 运营可以打开任意任务详情，但不显示编辑、重试、撤回、删除、重新提交或管理员审批操作。
- VIS 标准源保持只读可见；系统设置页保持只读说明，不提供保存能力。

### 任务详情

新增“运营补充评价”面板：

- 所有角色均可查看历史评价。
- 历史评价按提交轮次和创建时间展示评价人、关注点、正文与时间。
- 只有运营显示提交表单。
- 表单包含可选“关注点”和必填“补充评价”文本域。
- 提交期间禁用按钮并显示提交状态；成功后清空表单并重新读取详情。
- 接口错误显示在详情页现有错误区域，不改变当前任务视图。

## 权限实现边界

- `Role`、`ClientRole`、`StoredSession` 和 `ReviewSession` 同步加入“运营”。
- 登录接口允许“运营”，并让运营使用 `REVIEW_ACCESS_CODE`。
- 现有创建、编辑、Frame、AI 审核、重提和删除接口继续通过 `assertRole`/`assertTaskPermission` 拒绝运营。
- 运营评价接口不使用管理员绕过语义，而是显式判断角色等于“运营”。
- 任务读取接口在服务端执行可见性检查：设计师仅能读取本人任务，运营和管理员能读取全部任务。前端默认筛选与服务端权限保持一致。

## 错误处理

- 无效运营口令沿用“访问口令错误”。
- 非运营提交评价返回“当前身份无权提交运营补充评价”。
- 空评价返回“请输入运营补充评价”。
- 任务不存在返回“任务不存在”。
- 数据库写入失败沿用现有持久存储错误转换，不产生半条评价或任务状态变化。

## 测试策略

### 单元与 API 测试

- 会话标准化接受运营角色，并拒绝未知角色。
- 角色切换辅助函数为运营恢复普通访问口令。
- 登录接口接受运营使用普通访问口令，管理员仍必须使用独立口令。
- 运营可以读取任务列表和任务详情。
- 设计师任务列表不包含他人任务，直接读取他人任务详情返回 403。
- 运营可以追加包含当前轮次、姓名、正文和关注点的评价。
- 评价提交前后任务状态、AI 分数、更新时间和最终结论完全一致。
- 连续提交生成两条记录，不覆盖历史。
- 空评价返回 400。
- 设计师和管理员提交运营评价返回 403。
- 运营创建、修改、删除、审批或重试任务继续返回 403。

### 前端与浏览器验证

- 登录页显示设计师、运营和管理员三个角色。
- 运营可使用普通口令进入工作台并看到全部任务。
- 运营工作台不显示新建任务按钮。
- 运营详情页只显示补充评价表单，不显示任务管理操作。
- 提交评价后立即显示新记录，刷新页面后记录仍存在。
- 设计师重新登录后能看到运营评价，但看不到提交表单。
- 管理员使用 `emke666` 登录成功，并保持管理员审批能力。
- 退出登录和重新选择角色正常工作。

## 部署与验收

1. 测试、TypeScript 检查、生产构建和 React Doctor 差异检查通过。
2. 将源代码提交并直接推送到 `main`。
3. 在 Vercel Production 中将 `REVIEW_ADMIN_ACCESS_CODE` 作为敏感变量设置为 `emke666`。
4. 重新部署生产环境。
5. 验证 `/api/health` 仍为 `storageMode=postgres`、`durableStorage=true`，并在生产浏览器完成三种角色的关键流程。
6. 使用 Supabase 查询确认运营评价已持久化，且 `emke_design_review_store` 继续启用 RLS，并未授予 `anon` 或 `authenticated` 访问权限。

验收完成时，设计师任务不会因角色扩展而改变原有流程；运营评价为独立、可追溯、只追加的补充信息；管理员权限继续由专用口令保护。
